# AI 작업 효율 정책

이 문서는 Agent 작업의 토큰과 반복 실행을 줄이면서 제품 검증 수준을 유지하기 위한 실행
규칙을 정의한다. 정확성, 보안, 데이터 무결성은 절약보다 우선한다.

## 컨텍스트 수집

- 먼저 `rg`로 위치를 찾고 필요한 파일과 줄 범위만 읽는다.
- 전체 파일, 전체 diff, 대형 JSON, benchmark 원본 로그를 반복해서 출력하지 않는다.
- 실패 로그는 테스트 이름, 오류 핵심, stack의 관련 프레임과 마지막 30~50줄을 우선 수집한다.
- 이미 확인한 계약과 변경되지 않은 모듈은 다시 읽지 않는다.

## 구현과 검증 순서

1. 구현 중에는 변경 모듈의 관련 unit/component test만 실행한다.
2. UI 변경은 개발 중 영향받는 Playwright spec 또는 viewport project만 실행한다.
3. 통합 시 영향받는 frontend 또는 Rust test 묶음과 lint/typecheck/clippy를 실행한다.
4. 코드가 확정된 뒤 전체 test suite와 `npm run test:e2e`를 한 번 실행한다.
5. release/NSIS build와 최종 GUI screenshot은 모든 코드 수정이 끝난 뒤 한 번 실행한다.
6. 최종 build 뒤 코드가 바뀌면 영향 테스트와 build만 다시 실행하고 이유를 기록한다.

전체 검증 실패 시 실패한 범위만 수정·재실행하고, 매 수정마다 전체 suite를 반복하지 않는다.
서로 독립적인 검사만 병렬 실행하며 package manager, formatter, lockfile 변경은 동시에 실행하지
않는다.

## Agent와 리뷰

- 단일 소유자가 처리할 수 있는 작업에는 subagent를 추가하지 않는다.
- 기본 상한은 `주담당 1 + 협업 0~2 + 품질 1`이며 실제 독립 작업이 있을 때만 병렬화한다.
- 리뷰 지적은 가능한 한 모아서 한 번에 수정하고 한 번에 재검토한다.
- Agent handoff는 변경 파일, 테스트 결과, 결함과 남은 위험만 요약하고 원본 로그를 복제하지 않는다.

## 사용자 보고

- 중간 보고는 시작, 범위나 위험의 중요한 변경, 장기 작업 상태, 완료 시점에만 한다.
- 명령 출력 전체를 전달하지 않고 PASS/FAIL, 개수, 핵심 오류와 산출물 경로만 알린다.
- 같은 사실과 계획을 반복 설명하지 않는다.
- 최종 보고는 구현, 검증, 실행 경로, BLOCKED 항목을 우선한다.

## 문서와 산출물

- 구현 중 문서를 계속 고치지 않고 계약 변경 또는 최종 결과가 확정됐을 때 묶어서 갱신한다.
- benchmark 원본은 artifact에 저장하고 대화와 통합 문서에는 측정값과 판정만 요약한다.
- 생성 cache, 임시 fixture, 중간 screenshot은 최종 증거가 아니면 정리한다.

## 절약보다 우선하는 경우

다음 상황에서는 필요한 추가 분석과 재검증을 생략하지 않는다.

- 데이터 손상, 보안 권한, 경로 검증 또는 임시 파일 정리 문제
- document/session/query identity와 비동기 경쟁 조건
- 공통 DTO, format registry, query engine처럼 여러 모듈에 영향을 주는 변경
- release에서만 나타나는 native, installer, clipboard 또는 WebView 문제
- 재현되지 않는 실패나 HIGH/MEDIUM 독립 리뷰 결함

이 정책은 테스트 수나 완료 기준을 줄이는 규칙이 아니다. 동일한 증거를 더 적은 반복과 더 작은
출력으로 수집하는 규칙이다.
