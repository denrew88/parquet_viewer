# Phase 8 UX 설계

- 상태: 설계 확정, 구현 대기

## 1. 목표와 사용자 모델

Phase 8은 다음 두 수준의 병렬 작업을 명확히 구분한다.

- **여러 프로그램 창:** 실행 파일을 여러 번 실행하면 각각 별도 프로세스와 창이 열린다. 각 창은 세션, 열린 파일, 오류, 선택, 복사 작업을 공유하지 않는다.
- **한 프로그램 안의 여러 파일:** 한 창에서는 파일마다 하나의 문서 탭을 만들고 탭을 전환해 탐색한다.

MVP는 한 프로세스 안에서 여러 native window를 관리하는 모델이 아니라 **프로세스당 창 하나**로 한다. 기존 single-instance 전달과 창 강제 포커스는 사용하지 않는다. 파일 연결로 파일을 열면 새 프로세스의 새 창이 열리고, OS가 한 번의 실행에 여러 경로를 전달하면 그 창의 여러 문서 탭으로 연다.

창을 닫으면 그 창에 속한 모든 파일 세션과 진행 중 작업을 정리한다. 읽기 전용 제품이므로 저장 확인이나 다른 창에 영향을 주는 확인은 표시하지 않는다. 탭을 다른 창으로 보내기, 탭을 떼어 새 창 만들기, 창 간 상태 공유는 후속 범위다.

## 2. 화면 정보 구조

위에서 아래 순서를 다음과 같이 정한다.

1. 앱 툴바: 앱 이름, 파일 열기 명령
2. **문서 탭 바:** 열린 CSV/Parquet 파일 전환과 닫기
3. **뷰 탭 바:** 활성 문서의 `Data`, `Schema`, `Metadata` 전환
4. 활성 문서 작업 영역
5. 활성 문서 상태 표시줄

문서 탭과 뷰 탭은 서로 다른 배경, 높이, label을 사용한다. 접근성 이름도 각각 `Open files`, `File views`로 구분한다. 현재 툴바의 파일 chip은 제거하거나 활성 문서 경로 표시로 축소하고, 파일 식별의 주 책임은 문서 탭에 둔다.

파일별로 다음 UI 상태를 독립 보존한다.

- 활성 `Data`/`Schema`/`Metadata` 뷰
- 현재 페이지와 스크롤 위치
- 논리 선택의 anchor, active cell, 직사각형 범위
- 컬럼 너비, 숨김, 검색
- 로딩, 행 수 계산, 오류와 재시도 상태

탭 전환은 세션을 닫거나 데이터를 다시 열지 않는다. 백그라운드의 행 수 계산은 계속할 수 있고, 탭을 닫을 때 그 문서의 작업을 취소한다. 상태 표시줄과 오류 banner는 활성 문서의 것만 본문에 표시하며, 비활성 문서의 로딩 또는 오류는 탭의 작은 상태 아이콘으로 알린다.

## 3. 셀 컨텍스트 메뉴

### 3.1 여는 방법

- 데이터 셀에서 마우스 오른쪽 버튼
- 그리드에 focus가 있을 때 `Shift+F10`
- 키보드의 Context Menu 키

MVP는 데이터 셀에만 제공한다. 행 번호, 컬럼 header, 그리드 corner의 전용 컨텍스트 메뉴는 후속 범위로 둔다. 브라우저 기본 컨텍스트 메뉴는 데이터 그리드 영역에서만 막고, 검색 input이나 다른 control에서는 유지한다.

### 3.2 선택과 우클릭의 관계

- 우클릭한 셀이 현재 선택 직사각형 **안**에 있으면 기존 선택, anchor, active cell을 그대로 유지한다. 메뉴 명령은 전체 선택 범위에 적용하고 우클릭 셀은 별도의 `context target`으로만 기억한다.
- 우클릭한 셀이 현재 선택 **밖**에 있으면 그 셀을 단일 셀로 새로 선택한 뒤 메뉴를 연다. anchor와 active cell 모두 우클릭한 셀이 된다.
- `Shift`, `Ctrl`, `Command`를 누른 우클릭은 선택을 확장하거나 토글하지 않는다.
- 로딩 placeholder처럼 값이 아직 없는 셀도 선택은 가능하지만, 값 자체가 필요한 메뉴 항목은 비활성화한다.
- 메뉴를 `Escape`로 닫으면 메뉴를 열기 전 선택을 유지하고 그리드 focus를 복원한다.

이 규칙으로 범위를 선택한 뒤 그 안에서 우클릭해 `Copy`하는 표준 스프레드시트 동작과, 선택 밖의 셀을 우클릭해 그 셀에 대한 명령을 실행하는 동작을 모두 보장한다.

### 3.3 MVP 메뉴 항목과 활성 조건

| 항목 | 동작 | 활성 조건 |
| --- | --- | --- |
| `Copy` | 현재 논리 선택 범위를 기존 TSV 복사 파이프라인으로 복사 | 유효한 선택이 있고 복사 작업이 진행 중이지 않음 |
| `Copy with column headers` | 현재 선택 범위 위에 해당 컬럼 이름을 추가해 TSV 복사 | 유효한 선택, 최소 1개 컬럼, 복사 작업이 진행 중이지 않음 |
| `Copy cell value` | context target 한 셀의 표시 값을 복사 | target 값이 로드됨. null은 기존 TSV/null 정책을 적용 |
| `View full value` | 기존 전체 값 확인 UI를 target 셀로 엶 | target 값이 로드됨 |

`Copy`에는 오른쪽에 `Ctrl+C` 또는 macOS의 `Cmd+C` 단축키 hint를 표시한다. hard limit을 넘는 선택은 메뉴 자체를 숨기지 않고, 실행 시 기존 제한 메시지를 표시한다. 이미 복사 중이면 복사 항목을 disabled 처리하고 기존 툴바의 취소 동작은 유지한다.

MVP 메뉴에 데이터 변경, 붙여넣기, 행 삭제, 필터링, 정렬, 내보내기는 넣지 않는다. 후속 후보는 `Select row`, `Select column`, `Copy row`, `Copy column name`, `Filter by value`이며 필터 기능이 제품 범위에 들어오기 전에는 노출하지 않는다.

### 3.4 위치와 viewport collision

- 포인터로 열면 기준점은 `clientX`, `clientY`다.
- 키보드로 열면 active cell의 화면 사각형 아래쪽 시작점에 붙인다. active cell이 가상화로 unmount된 경우 먼저 최소 거리로 scroll한 뒤 셀이 mount된 다음 연다.
- 기본 위치는 기준점의 오른쪽 아래다. 메뉴의 실제 크기를 측정한 뒤 오른쪽 공간이 부족하면 왼쪽으로, 아래 공간이 부족하면 위로 뒤집는다.
- 최종 위치는 WebView viewport의 네 변에서 최소 8 CSS px 안쪽으로 clamp한다.
- 메뉴 높이가 사용 가능한 영역보다 크면 최대 높이를 설정하고 메뉴 내부만 세로 scroll한다. 문서와 그리드의 scroll 위치를 메뉴 배치 때문에 바꾸지 않는다.
- 창 resize, 화면 배율 변경, 그리드 scroll, 활성 문서 변경 시 열린 메뉴를 닫는다. 메뉴가 오래된 가상 셀 위치에 남아 있지 않게 한다.
- 메뉴는 portal로 앱 shell 최상단에 렌더링하고 기존 grid/header/loading overlay보다 높은 명시적 z-index를 사용한다.

### 3.5 키보드와 접근성

- 메뉴 컨테이너는 `role="menu"`, 항목은 `role="menuitem"`을 사용한다.
- 메뉴가 열리면 첫 번째 활성 항목에 focus한다. `ArrowUp`/`ArrowDown`으로 순환하고, `Home`/`End`로 처음/마지막 활성 항목에 이동한다.
- `Enter` 또는 `Space`로 실행한다. `Escape`는 실행 없이 닫고 그리드 focus를 복원한다.
- `Tab` 또는 `Shift+Tab`은 메뉴를 닫고 다음 또는 이전 일반 focus 대상으로 이동하게 해 focus trap을 만들지 않는다.
- disabled 항목은 시각적 대비 외에 `aria-disabled="true"`로 전달하며 실행되지 않는다.
- 메뉴가 열린 동안 그리드 화살표 선택 handler는 동작하지 않는다. 메뉴를 닫은 뒤 기존 선택 키보드 계약이 재개된다.
- 메뉴 label은 화면에 보이는 명령명 자체로 충분히 설명하며, 아이콘만 단독으로 사용하지 않는다.

## 4. 문서 탭

### 4.1 열기와 중복

- 파일 대화상자는 CSV/Parquet 다중 선택을 허용한다.
- drag and drop과 시작 인자도 여러 지원 파일을 입력 순서대로 각각 탭으로 연다.
- batch 열기에서는 첫 번째로 성공한 파일을 활성화하고 나머지는 background 탭으로 연다. 비동기 완료 순서 때문에 활성 탭이 계속 바뀌지 않게 한다.
- 정규화한 같은 경로가 이미 해당 창에 열려 있으면 중복 세션을 만들지 않고 기존 탭을 활성화한다.
- 지원 파일과 실패 파일이 섞이면 성공 파일은 탭으로 유지하고 실패 목록은 하나의 요약 오류로 알린다.
- 8개 동시 열린 문서는 일반 사용 검증 기준이며 최대치가 아니다. 창당 방어 상한은 64개이고, 한 번의 batch는 최대 32개다. 65번째 파일은 열지 않고 현재 열린 수와 제한을 설명한다.

### 4.2 탭 표시

- 탭 기본 label은 파일명이다. 같은 파일명이 둘 이상이면 구분 가능한 가장 가까운 상위 폴더명을 함께 표시한다.
- hover tooltip에는 정규화한 전체 경로를 표시한다.
- 활성 탭은 색상뿐 아니라 하단 경계와 `aria-selected`로 구분한다.
- 탭의 닫기 명령은 Lucide `X` 아이콘과 tooltip/accessible label `Close <file name>`을 사용한다.
- 파일을 여는 중에는 spinner, 오류가 있는 비활성 탭에는 경고 아이콘을 표시한다. 이 아이콘이 탭 너비를 바꾸지 않도록 공간을 고정한다.
- 창 title은 `<active file> - Data Viewer`로 하고 파일이 없으면 `Data Viewer`로 한다.

### 4.3 활성 전환과 키보드

- 클릭으로 탭을 활성화한다.
- 문서 tablist에 focus가 있을 때 `ArrowLeft`/`ArrowRight`, `Home`/`End`로 이동하며 선택도 함께 전환한다.
- `Ctrl+Tab`과 `Ctrl+Shift+Tab`은 작업 영역 어디서든 다음/이전 문서를 순환한다.
- `Ctrl+W`는 활성 문서를 닫는다. 마지막 문서를 닫으면 빈 workspace로 돌아간다.
- 탭 focus는 roving `tabIndex`를 사용하고, 뷰 탭의 키보드 이동과 별도 tablist로 관리한다.

뷰 전환 상태는 문서마다 보존한다. 예를 들어 A 파일의 `Metadata`에서 B 파일의 `Data`로 갔다가 A로 돌아오면 A의 `Metadata`가 다시 보인다.

### 4.4 닫기

- 각 탭의 닫기 아이콘 또는 `Ctrl+W`로 닫는다. 읽기 전용이고 dirty 상태가 없으므로 확인 dialog를 표시하지 않는다.
- 활성 탭을 닫으면 오른쪽 이웃을 우선 활성화하고, 없으면 왼쪽 이웃을 활성화한다.
- 비활성 탭을 닫으면 현재 활성 탭을 바꾸지 않는다.
- 열기, 행 수 계산, 페이지 조회, 복사가 진행 중인 탭을 닫으면 그 문서에 속한 작업을 취소하고 session, handle, index, cache를 해제한다.
- 중간 클릭 닫기는 익숙한 동작이지만 오작동 검증 범위를 줄이기 위해 MVP 후속으로 둔다.

dirty marker와 저장 확인은 없다. 미래에 편집 기능이 추가될 때만 dirty 상태를 별도 설계한다.

### 4.5 overflow와 reorder

- 문서 탭 바는 한 줄을 유지하고 가로 overflow를 사용한다. 탭을 여러 줄로 감싸 작업 영역 높이를 바꾸지 않는다.
- 좌우 scroll 아이콘 버튼을 제공하며 overflow가 없을 때 disabled 처리한다.
- 활성 탭으로 전환하거나 새 탭이 생기면 그 탭 전체가 보이는 최소 거리만 자동 scroll한다.
- 800x600에서도 열기 버튼, 문서 탭, 뷰 탭이 서로 겹치지 않아야 한다. 긴 label은 고정된 최대 너비에서 ellipsis 처리하고 tooltip으로 전체 식별 정보를 제공한다.
- **MVP에서는 탭 순서를 연 순서로 고정하고 drag reorder를 제공하지 않는다.** 탭 reorder, 모든 탭 목록 menu, 고정 탭은 후속 범위다.

## 5. 다중 프로세스와 다중 탭의 세부 규칙

| 사용자 동작 | MVP 결과 |
| --- | --- |
| 앱 아이콘 또는 exe를 두 번 실행 | 서로 독립적인 프로세스와 창 두 개 |
| 파일을 OS에서 더블클릭 | 새 프로세스/창에서 해당 파일 탭 활성화 |
| 한 번의 OS 실행에 여러 경로 전달 | 새 창 하나에 여러 문서 탭 |
| 기존 창의 Open file에서 여러 파일 선택 | 그 창에 여러 문서 탭 추가 |
| 기존 창에 여러 파일 drop | 그 창에 여러 문서 탭 추가 |
| 같은 창에서 이미 열린 경로 다시 열기 | 기존 탭 활성화 |
| 다른 창에서 이미 열린 경로 열기 | 허용. 별도 읽기 전용 세션 |
| 한 창 닫기 | 그 프로세스 세션만 정리, 다른 창은 영향 없음 |

각 프로세스가 같은 파일을 동시에 읽는 것은 허용한다. 앱은 계속 파일을 읽기 전용으로 열며 창 간 lock이나 변경 동기화를 도입하지 않는다.

## 6. MVP와 후속 범위

### Phase 8 MVP

- 기존 선택 계약과 연결된 셀 컨텍스트 메뉴
- `Copy`, `Copy with column headers`, `Copy cell value`, `View full value`
- 키보드 메뉴 접근과 viewport collision 처리
- single-instance 동작 제거와 독립 프로세스 동시 실행
- 8개 이상 일반 사용과 최대 64개의 파일 문서 탭, 32개 batch dialog/drop/startup open
- 파일별 뷰, 페이지, scroll, 선택, 컬럼 설정, 오류 상태 보존
- 탭 닫기, 활성 전환, 키보드 순환, 한 줄 overflow
- 탭 닫기와 창 종료 시 session/task/cache 정리

### 후속 범위

- 행/컬럼 header 컨텍스트 메뉴와 추가 선택 명령
- filter/sort/export 등 새로운 데이터 명령
- 탭 drag reorder, middle-click close, 전체 탭 목록, pin
- 탭을 새 창으로 떼기, 창 간 이동, single-process multi-window
- 창/탭 세션 복원과 최근 파일
- 편집 기능이 생긴 경우 dirty marker와 저장 확인

## 7. UX 검증 초점

- 범위 안/밖 우클릭에서 선택 좌표가 위 계약대로 유지 또는 교체되는지 확인
- 포인터, `Shift+F10`, Context Menu 키로 같은 명령과 활성 조건이 제공되는지 확인
- 메뉴가 1440x900, 1024x768, 800x600의 네 모서리 셀에서 viewport를 벗어나지 않는지 geometry 측정
- menu focus 이동, 실행, `Escape`, `Tab` 이후 focus 복원 검증
